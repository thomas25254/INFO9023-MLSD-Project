# HearEdit — ML Pipeline Documentation

## Overview

The HearEdit ML pipeline is a **Vertex AI Kubeflow Pipelines (KFP)** pipeline that automates the training and evaluation of a speaker verification model. It is designed to be run periodically (for example, monthly) to retrain the model on new data and automatically update the decision threshold used in production by the API.

### Why a pipeline?

Without a pipeline, each step of the training cycle (data preparation, training, evaluation, threshold calculation) would have to be executed manually, in the correct order, on the correct machine. The Vertex AI pipeline automates this entire process: it orchestrates the steps, manages dependencies between them, and runs entirely in the cloud without manual intervention.

### Trained model

The pipeline fine-tunes an **ECAPA-TDNN** (Emphasized Channel Attention, Propagation and Aggregation Time Delay Neural Network) model from the SpeechBrain library, pre-trained on the VoxCeleb dataset. This model extracts 192-dimensional voice embeddings from an audio signal, enabling the comparison of voices via cosine similarity.

Training uses an **ArcFace loss** (Additive Angular Margin Loss), a loss function specifically designed to learn embeddings that are well-separated in angular space—which improves the model’s ability to distinguish between different speakers.

## Pipeline DAG

```
data_preparation (CPU)
        |
        |── train_split ──┐
        |── val_split   ──┤──► training (GPU T4)
        |── test_split ───┤         |
                          |         |── model ──► evaluation (CPU)
                          |         |── model ──► compute_threshold (CPU)
                          └─────────┘
```

The arrows represent the dependencies between the steps. `training` can only start once `data_preparation` is complete. `evaluation` and `compute_threshold` can only start once `training` is complete.

## Components

### 1. `data_preparation`

**Machine:** CPU (default)

**Description:**

This component is the entry point of the pipeline. It downloads the LibriSpeech dataset from Google Cloud Storage, identifies all `.flac` audio files, and divides them into three distinct sets: training, validation, and test.

The split is done **by speaker** (not by audio file) to ensure that no speaker present in the test set was encountered during training. This guarantees an unbiased assessment of the model’s generalization performance.

The splits are saved as JSON files via the KFP artifact system, allowing subsequent components to access them directly from GCS without having to re-download the data.


**Input:**
- `gcs_dataset_uri` — GCS path to the dataset (e.g., `gs://mlops-2026-dataset-bucket/train-clean-100/train-clean-100/`)

**Output artifacts:**
- `train_split` — JSON file containing 80% of the utterances per speaker, with the audio path and speaker ID
- `val_split` — JSON file containing 10% of the utterances per speaker, used to monitor training
- `test_split` — JSON file containing 10% of the utterances per speaker, reserved for final evaluation

**Split strategy:** 80/10/10 per speaker. Speakers with fewer than 3 utterances are excluded because they cannot be distributed across the three sets.

---

### 2. `training`

**Machine:** `n1-standard-4` + NVIDIA Tesla T4 GPU (15 GB RAM)

**Description:**

This component is the core of the pipeline. It downloads audio files from GCS, loads the splits from `data_preparation`, and fine-tunes SpeechBrain’s ECAPA-TDNN encoder.

At each epoch, the model is trained on the training set using ArcFace loss, then evaluated on the validation set. Validation accuracy is calculated by comparing the embeddings to the normalized weights of the ArcFace layer using cosine similarity. The best checkpoint (the one with the highest validation accuracy) is saved to GCS.

The T4 GPU is used exclusively for this step because training is the most computationally intensive operation.


**Inputs:**
- `train_split`, `val_split` — KFP Dataset artifacts generated by `data_preparation`
- `gcs_dataset_uri` — GCS path for downloading audio files locally within the container
- `epochs` (default: 5) — Number of complete passes over the training set
- `batch_size` (default: 8) — Number of utterances processed simultaneously

**Output artifacts:**
- `model` — KFP folder containing:
  - `encoder.pt` — Weights of the fine-tuned ECAPA-TDNN model (embedding model only)
  - `speaker_labels.pt` — Speaker ID to integer index mapping, required for inference

**Model architecture:**
- Base: `speechbrain/spkrec-ecapa-voxceleb` (pre-trained on VoxCeleb)
- Loss: ArcFace (scale s=30.0, margin m=0.5)
- Optimizer: Adam with two parameter groups:
  - ECAPA encoder LR: `3e-6` (very low to avoid destroying pre-trained weights)
  - ArcFace head LR: `5e-4` (higher to quickly learn the speaker mapping)
- Embedding dimension: 192

**Metrics logged in Vertex AI:**
- `train_loss_epoch_{n}` — Average ArcFace loss on the training set at epoch n
- `val_acc_epoch_{n}` — Classification accuracy on the validation set at epoch n
- `best_val_acc` — Best validation accuracy achieved across all epochs
- `num_speakers` — Number of distinct speakers in the training set

---

### 3. `evaluation`

**Machine:** CPU (default)

**Description:**



This component measures the actual performance of the fine-tuned model on data it has never seen during training (the test set). This is a **speaker verification** evaluation—not a classification task.

For each speaker in the test set, the component extracts the embeddings of all their utterances using the fine-tuned model. It then calculates:
- **Intra-speaker similarities**: cosine similarities between utterances from the same speaker (these values should be high)
- **Inter-speaker similarities**: cosine similarities between utterances from different speakers (these values should be low)

Based on these two distributions, it calculates the ROC curve and derives the **EER (Equal Error Rate)** — the error rate at the point where the false positive rate equals the false negative rate. A low EER indicates a high-performing model.

**Inputs:**
- `test_split` — KFP Dataset artifact from `data_preparation`
- `model` — KFP Model artifact from `training`
- `gcs_dataset_uri` — GCS path to download the audio files

**Metrics logged in Vertex AI:**
- `eer` — Equal Error Rate on the test set (between 0 and 1; the lower, the better)
- `auc` — Area under the ROC curve (between 0 and 1; the higher, the better)
- `num_test_speakers` — Number of speakers used for evaluation

---

### 4. `compute_threshold`

**Machine:** CPU (16 GB RAM)

**Description:**

This component calculates the optimal cosine similarity threshold to use in production. This threshold is the value below which two audio segments are considered to come from different speakers, and above which they are considered to come from the same speaker.

The key difference from the `evaluation` component is as follows:
- `evaluation` measures the model’s **performance** on an unseen test set → quality metric
- `compute_threshold` calculates the **production threshold** on a sample of training data → operational parameter

The threshold calculation follows the same EER logic as the evaluation: we seek the threshold that simultaneously minimizes false positives (two different speakers confused) and false negatives (same speaker not recognized).

Due to memory constraints, the calculation is performed on a random sample of 3,000 files. This number is sufficient to obtain a reliable threshold—the literature shows that EER converges rapidly with a sufficient number of intra- and inter-speaker pairs.

**Inputs:**
- `gcs_dataset_uri` — GCS path to the data (3,000 randomly selected files)
- `gcs_threshold_output_uri` — GCS path to save the output file
- `model` — KFP Model artifact from `training`


**Output:**
The `threshold_pipeline.json` file saved to `gs://hearedit-models/artifacts/threshold_pipeline.json` and loaded by the API in production:

```json
{
  "threshold": 0.2670,
  "eer": 0.0079,
  "auc": 0.9984
}
```

**Metrics logged in Vertex AI:**
- `eer` — Equal Error Rate on the training set
- `auc` — Area under the ROC curve
- `threshold` — Cosine similarity threshold used for production

## Run the Pipeline

### Prerequisites

```bash
# Activate the virtual environment
source ~/Documents/2025-2026/Q2/MLSD/Project/.venv/bin/activate

# Authenticate on GCP (required only once per session)
gcloud auth application-default login
```

### Launch

```bash
cd src/pipeline
python run_pipeline.py
```

This command performs two actions:
1. **Compilation** — Serializes the pipeline into `hearedit_pipeline.json` (YAML/JSON KFP format)
2. **Submission** — Sends the job to Vertex AI Pipelines, which orchestrates the execution in the cloud

### Monitoring

Go to **GCP Console → Vertex AI → Pipelines → Runs** (region: `europe-west1`) to monitor the pipeline execution. Each component displays its real-time logs, metrics, and status (running, success, failure).

## Docker Image

Each KFP component runs in an isolated Docker container. This means that the Python code for each component is **injected at runtime** by KFP into the container, rather than being copied into the image. The Docker image contains only the Python dependencies.

**Registry:** `europe-west1-docker.pkg.dev/info9023-project-hearedit/hearedit-pipeline/pipeline-base:latest`

**Main dependencies:**
- `torch`, `torchaudio`, `speechbrain` — Deep learning and audio processing
- `kfp`, `google-cloud-aiplatform`, `google-cloud-storage` — Vertex AI and GCS integration
- `scikit-learn`, `soundfile`, `numpy` — Statistical calculations and audio playback

**Rebuild and push the image (only if dependencies change):**

```bash
cd src/pipeline
export DOCKER_HOST=unix:///var/run/docker.sock
docker build --platform linux/amd64 -t pipeline-base:latest .
docker tag pipeline-base:latest \
    europe-west1-docker.pkg.dev/info9023-project-hearedit/hearedit-pipeline/pipeline-base:latest
docker push \
    europe-west1-docker.pkg.dev/info9023-project-hearedit/hearedit-pipeline/pipeline-base:latest
```

> **Note:** The image is approximately 9–10 GB in size due to PyTorch and SpeechBrain dependencies. The build takes about 10–15 minutes.

## Output Artifacts

After the pipeline has run successfully, the following artifacts are available on GCS:

| Artifact | GCS Path | Description |
|---|---|---|
| Fine-tuned encoder | `gs://hearedit-models/pipeline_root/<run_id>/training_<id>/model/encoder.pt` | PyTorch weights of the ECAPA-TDNN model |
| Speaker labels | `gs://hearedit-models/pipeline_root/<run_id>/training_<id>/model/speaker_labels.pt` | Speaker ID to index mapping |
| Production threshold | `gs://hearedit-models/artifacts/threshold_pipeline.json` | EER threshold for the API |

The `threshold_pipeline.json` is the only artifact directly consumed by the API in production. It is overwritten with each new pipeline run, allowing for automatic threshold updates without redeploying the API.
